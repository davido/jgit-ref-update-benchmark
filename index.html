JGit Ref-Update Benchmark: RefUpdate vs BatchRefUpdate(N=1)

Empirical JMH measurement of two paths to a single ref update — on JGit's RefDirectory backend, across CREATE and packed/loose UPDATE scenarios up to two million refs.

by David Ostrovsky— Gerrit & JGit maintainer

The Question

Why this benchmark exists in the first place.

The Gerrit pull-replication plugin had a half-apply bug in its multi-ref batch-apply-object endpoint (Issue 456926963): when ref N failed pre-flight on a missing parent, refs 1..N−1 had already landed. The structural fix was to route the multi-ref endpoint through one atomic BatchRefUpdate.

The earlier proposal routed both single-ref and multi-ref endpoints through BatchRefUpdate. A brain-storming session worried this was a performance regression for the single-ref code path — and pushed for a bifurcation that keeps single-ref on RefUpdate and only routes multi-ref through BatchRefUpdate. The bifurcation shipped as a 4-commit chain in the GerritForge fork.

The unasked question: is BatchRefUpdate.execute() with a single ReceiveCommand actually slower than RefUpdate.update() on real-sized repositories? Nobody had measured it.

This benchmark answers that question with JMH on the upstream JGit, on the RefDirectory backend, across four ref-count scales (1K, 10K, 1M, 2M) and three scenarios (CREATE, packed-start UPDATE, loose-start UPDATE).

BatchRefUpdate(N=1) is as fast as — or faster than — RefUpdate at every reliably-measured point. Every UPDATE favours BRU (0.54–0.71), and CREATE is parity-to-BRU-favoured at 1M/2M (0.87 / 0.93). The 1K/10K CREATE rows are noise (error bars as large as the values). An earlier throwaway run hinted at a CREATE "inversion" (BRU ~1.67× slower at 1M); it did not reproduce once the benchmark's ref-selection bug was fixed and the run was given real forks and iterations.

Methodology

JMH on upstream JGit. Complements Luca Milanesio's GetRefsBenchmark (read path) with the write-path counterpart.

Harness: JMH 1.37, @BenchmarkMode(Mode.AverageTime), @OutputTimeUnit(MICROSECONDS). 2 forks × (2×100 ms warmup + 3×5 s measurement); ± is the 99.9 % CI.

JDK: 21.0.7 (Zulu) — JDK 25 trips the JGit Maven build's gmavenplus-plugin.

Source: a new RefUpdateBenchmark.java in org.eclipse.jgit.benchmarks proposed for upstream JGit. Mirrors the existing GetRefsBenchmark pattern (@State, @Setup, @Param) faithfully.

Native git reference

Canonical git (C) and libgit2 numbers as comparison baselines for JGit.

Native git has no in-process API, so a perfectly fair head-to-head with JGit's in-process measurement is impossible. Three useful baselines, each measuring a slightly different thing:

  • Per-call git update-ref — one fork per ref. Dominated by fork() + execve() cost (~10-50 ms on macOS). Useful only as a "what does shelling out cost?" data point.
  • Bulk git update-ref --stdin — one fork, N commands batched. Closest fair comparison to JGit's BatchRefUpdate with N=large; divide total time by N for amortised per-op cost.
  • libgit2 via pygit2 — in-process via Python FFI. A different C implementation of git's on-disk format, not canonical git, but reading/writing the same files. Per-op cost is in the same ballpark.

Numbers below are on a Darwin x86_64 laptop, git 2.54.0, pygit2 1.19.3 / libgit2 1.9.4, against a 100,000-ref bench repo populated via git update-ref --stdin + git pack-refs --all.

What the source shows

Why the two N=1 paths are directly comparable — and what's left to profile.

PackedBatchRefUpdate.execute short-circuits at pending.size() == 1 — explicit in the JGit source:

// Single-ref updates are always atomic, no need for packed-refs.
super.execute(walk, monitor, options);

So a 1-command BatchRefUpdate on RefDirectory skips the packed-refs locking/rewriting machinery and falls through to the base BatchRefUpdate.execute, which loops once and creates a single RefUpdate via BRU.newUpdate(cmd). Both paths therefore bottom out in the same RefDirectoryUpdate.doUpdate for the write itself — which is what makes the N=1 comparison meaningful: any difference is wrapper, not the underlying write.

What we measured: that wrapper difference favours BatchRefUpdate(N=1) at every reliably-measured point, and the gap is larger for UPDATE (0.54–0.71×) than for CREATE (0.87–0.93× at 1M/2M). Why the wrapper costs differ — and why UPDATE differs more than CREATE — is not something this benchmark establishes. Pinning it down (a candidate to check: the constructed RefUpdate's reflog settings vs a direct RefUpdate's refLogMessage = "" default) is left to a profiling/debug session.

TL;DR

BatchRefUpdate(N=1) is not slower. The bifurcation's performance fear isn't supported.

On RefDirectory, BatchRefUpdate(N=1) is as fast as — or faster than — RefUpdate.update() at every reliably-measured scale:

UPDATE (packed-start 0.63× at 1M, 0.54× at 2M; loose-start 0.71× at 10K): BRU clearly faster. Why the gap is larger for UPDATE than CREATE is not established here — left to a profiling session.

CREATE (0.87× at 1M, 0.93× at 2M): parity-to-BRU-favoured. At 1K/10K the error bars are as large as the values, so those aren't rankable — and the earlier "CREATE inverts to 1.67× slower at 1M" did not reproduce once the benchmark's ref-selection bug was fixed.

Implication for the pull-replication bifurcation that motivated this benchmark: it kept single-ref updates off BatchRefUpdate for fear of a per-N=1 slowdown. Measurement doesn't support that fear. The architectural-cleanliness case may stand on its own; the performance case does not.

The wider lesson: a single-fork JMH run over a flawed setup can manufacture a clean-looking trend. Forks, iterations, and a correct harness matter.
